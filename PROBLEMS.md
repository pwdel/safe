# LLM Coding Guardrail Problems and Task Plan

| Problem Area | Deduplicated Problem | Task List (Methodology) |
| --- | --- | --- |
| Context Management | Context drift in longer sessions (forgets prior constraints, repeats bad fixes) | Keep `experimental.chat.messages.transform` pre-compaction active; add pinned `CONSTRAINTS.md` injected as startup context; add a turn-level "constraint check" step in orchestrator prompt. |
| Code Correctness | Correct-looking code with broken logic, state handling, and edge cases | Route implementation through `team-implement` and mandatory `team-review`; require test-first task decomposition; block "done" responses unless targeted tests ran and output is attached. |
| Architecture | Weak architectural alignment (noisy, unidiomatic, hard-to-maintain patterns) | Create an architecture rubric checklist and run it in `team-review`; add a "no new pattern without rationale" rule in prompts; require explicit architecture impact note per change. |
| Scope Control | Unwanted scope creep (unrelated refactors/files/dependencies) | Add diff-scope hook to fail if changed files are outside approved paths; add dependency-change gate requiring explicit approval; enforce "small patch mode" in implementation prompts. |
| Task Definition | Instruction-following failures and ambiguous task framing | Standardize task intake template (goal, non-goals, file scope, acceptance tests); have orchestrator split large asks into subtasks before coding; reject execution when acceptance criteria are missing. |
| Tooling Reliability | Tool misuse and failed execution flow (wrong tool, repeated dead-end actions) | Add hook-based retry policy: after N repeated tool failures, switch to diagnose mode; require explicit tool-selection rationale for risky ops; surface concise failure summaries to orchestrator. |
| Quality Consistency | Inconsistent quality across runs/prompts | Pin model per agent role (`review` stable, `implement` capable); add deterministic formatting/lint/test pipeline; require second-pass review subagent before final output. |
| Productivity and ROI | High supervision overhead and low ROI | Add token/time budget per task and stop conditions; prefer boilerplate/refactor/docs tasks for automation; auto-log rework causes in `.context/` for weekly prompt/hook tuning. |
| Security and Safety | Security and safety risk (destructive commands, secrets exposure, unsafe actions) | Keep/expand `tool.execute.before` denylist and path checks; deny `.env` reads by default; enforce bash allowlist for subagents; require manual approval for network/write outside project. |
| Validation and Testing | Regression risk from incomplete validation | Add stop gate requiring test evidence for touched areas; require regression checklist in final response; trigger reviewer subagent for high-risk files before completion. |
| Orchestration Capability | Limited true multi-agent team orchestration in current OpenCode release | Use current `task`-based orchestrator/subagent pattern as interim; maintain a "ready to adopt" checklist for upstream Agent Teams once merged; track upstream issue/PR status monthly. |
| Platform and Access | Provider/platform usage constraints and portability pain | Document fallback runtime plan (`opencode` primary, `codex` fallback); keep auth/plugin setup scripted; maintain provider-neutral prompts and minimal tool-specific assumptions. |
